Design Elements

Grids

We use Bootstrap's 12-column responsive grid, on both desktop and mobile, and prebuilt grid classes for layouts that scale and stack based on device and viewport size. Columns allow us to adjust the placement and size of modules within layouts, making the best use of the space available at each breakpoint and maintaining hierarchy within layouts.

This is a sample two-column layout where the columns stack using the full width on mobile.

Spacing

The Mapzen UI is based on spacing in multiples of 6. Use this scale as a starting point when laying out pages and sizing elements.

Vertical Spacing

We have several responsive classes available to adjust vertical spacing on your page. Two of the most frequently used are .headroom and .footroom, which provide margin-top and margin-bottom spacing, respectively.

Both classes come in a few different sizes. For example: headroom, headroom-med, headroom-large, and headroom-extra-large

To add vertical space between page sections, a good place to start is .headroom-large

Spacing Utility Classes

If you need more fine-grained spacing control over your elements, we've incorporated a set of responsive-friendly spacing utility classes (based on Bootstrap v4 spacing) into the styleguide. Use these to modify the margins and/or padding around elements.

The format for spacing classes is {property}{sides}-{breakpoint}-{size} and allows for the following options:

{property}

  • m- margin
  • p- padding

{sides} (optional)

  • t- top
  • b- bottom
  • l- left
  • r- right
  • x- left and right
  • y- top and bottom
  • Leave blank to set all sides

{breakpoint} (optional)

  • sm
  • md
  • lg
  • Leave off -{breakpoint} to ignore breakpoints

{size}

  • 0- 0px
  • 1- 6px ($spacer * .25)
  • 2- 12px ($spacer * .5)
  • 3- 24px ($spacer)
  • 4- 36px ($spacer * 1.5)
  • 5- 72px ($spacer * 3)
  • auto- auto (for positioning fixed-width block content)

Examples of these classes 

    .mt-0 {
      margin-top: 0 !important;
    }

    .ml-1 {
      margin-left: 6px !important;
    }

    .px-2 {
      padding-left: 12px !important;
      padding-right: 12px !important;
    }

    .p-3 {
      padding: 24px !important;
    }

    // Small grid and above
    @media (min-width: 768px) {
      .m-sm-4 {
        margin: 36px !important;
      }

      .pb-sm-5 {
        padding-bottom: 72px !important;
      }
    }

    // Use with fixed-width block elements
    .mx-auto {
      margin-right: auto !important;
      margin-left: auto !important;
    }

Buttons

Use verbs on buttons as they are there to perform actions. Make it clear what the button does – users shouldn’t need to read any other text to understand what they do. Negative actions can be shorter, as the positive action makes their context clear. The button text should make sense in both these contexts, “Would you like to [button text]?” and “I would like to [button text]”? If a button leads to another screen, then the title of that following screen should match the button.

This is the primary button style for important calls to action. Two primary buttons cannot be used next to each other. If there are many buttons on a single page or screen, use the primary button sparingly for the most important actions on that page.

The `.btn-primary` class replaces the old `.btn-mapzen` class, which is now deprecated.

This is the secondary button style. Used next to a primary style if there are two buttons next to each other. If there are many buttons on a single page or screen, use the secondary button for the majority of buttons, so that the most important actions on a page (using the primary buttons) stand out.

This is the inverse button style. Use it on non-white backgrounds.

If a button should be full-width on mobile, add the .btn-flexible class.

A series of buttons can be placed next to each other, and spacing will be applied. This works for buttons that are in line as well as flexible buttons that will be stacked vertically on mobile.

This style is used for calls to action after paragraphs of text and draws less attention than a proper button.

Buttons require JavaScript to handle clicks and perform actions. If you are just using a button to link to another page, use an anchor tag.

Extra wide tables

Tables will attempt to fit in the space it's allotted. However, if a table’s content prevents it from fitting, then the table is allowed to scroll horizontally so that it cannot break the page layout. This is made possible by a <div class=".table-wrapper"> element that wraps the table element. If you do not provide this element yourself, we will automatically add one.

If you do not desire the horizotal scroll and wish to minimize its effect, consider the following content guidelines:

  • Avoid writing extremely long, unbreakable strings of text, such as very_long_variable_names. You can force long variable names to break by applying a .table-word-break class to the table element (see below; this is not available in Markdown).
  • Similarly, avoid showing full URL strings. URLs should be written as links with readable text labels.
  • Keep the number of columns to the minimum necessary to convey information. It may be desirable to present information across several tables instead of one large table.
  • Not all information is best presented as tables. Consider whether this data is tabular in nature and if there are other ways of presenting it. See the content guidelines for more information.
  • On mobile devices, this behavior may be necessary and unavoidable.

Breaking code blocks in tables

Currently, we do not implement word breaking or hyphenation, due to poor cross-browser support. However, it may be desirable in some situations to break code elements containing long_variable_names_such_as_this_one with minimal impact on readability. This is available by adding a .table-word-break class to the table element, and it takes effect only on screens smaller than 768 pixels wide (e.g. on mobile devices). It must be written in HTML manually; it is not available for tables that are generated through Markdown.

Alerts

Make it clear that an error has occurred (in language the user can understand), and that the user needs to do something. Buttons and links inside alerts should provide the user with a way to solve the problem.

Inform users of errors in a friendly way. Don’t make them feel bad for making a mistake. Also avoid exclamation marks – this can make the error look more significant than it really is.
